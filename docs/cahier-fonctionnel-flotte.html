1. Contexte & Objectifs

Le module web de Gestion de Flotte constitue le tableau de bord destiné aux responsables de société de transport (COMPANY_MANAGER) et aux fournisseurs (SUPPLIER) au sein de la plateforme MaliTrans.

Objectifs principaux

  • Visualiser et filtrer l'ensemble des chauffeurs rattachés à la société
  • Traiter les demandes d'inscription de chauffeurs en attente de validation
  • Valider les dossiers complets (garants + documents d'identité)
  • Consulter le dossier complet de tout chauffeur de la flotte
  • Suivre le statut d'avancement de chaque chauffeur dans le workflow d'activation
Contrainte de sécurité critique : Un manager ne peut voir et gérer QUE les chauffeurs de SA propre société. Cette restriction est appliquée côté backend de façon stricte et non contournable.

Périmètre du document

Ce cahier fonctionnel couvre exclusivement le module de gestion de flotte. Les modules Administration (ADMIN), espace Chauffeur et espace Client sont hors périmètre.

2. Acteurs & Rôles

Acteur Rôle système Cardinalité Accès au module flotte
Gestionnaire Société COMPANY_MANAGER 1 par société Accès complet (lecture + validation)
Fournisseur SUPPLIER Multiple par société Accès complet (lecture + validation) + gestion livraisons
Chauffeur CHAUFFEUR Multiple par société Aucun accès au module flotte
Administrateur ADMIN Global Gestion globale (hors périmètre ce document)
Note : Un SUPPLIER possède les mêmes droits qu'un COMPANY_MANAGER sur le module flotte. La différence réside dans ses capacités supplémentaires de gestion des livraisons.

4.2 File d'attente — Chauffeurs à Valider

GET /api/company/drivers/pending   Auth : JWT Bearer Token

Retourne la liste complète des chauffeurs en statut PENDING_COMPANY_VERIFICATION sans pagination (liste directe), triée par ID décroissant (les plus récents en premier).

Règles UI

  • Accessible depuis un badge cliquable dans la navigation principale
  • Chaque ligne dispose d'un bouton "Ouvrir le dossier" → redirection vers l'écran §4.3
  • Liste vide → afficher un message "Aucun chauffeur en attente de validation"
  • Rafraîchissement automatique suggéré toutes les 60 secondes (polling léger)

Organisation de l'écran dossier

Section 1 — Identité Chauffeur

  • Nom complet, téléphone, adresse
  • Type de véhicule
  • Matricule (si déjà généré)
  • Statut avec badge coloré
  • Document d'identité : bouton de téléchargement / aperçu PDF

Section 2 — Garants

  • Compteur : "X / 2 garant(s) renseigné(s)"
  • Pour chaque garant : nom, téléphone, adresse, relation
  • Document d'identité de chaque garant

Section 3 — Actions

  • Bouton "Valider le dossier" — visible uniquement si status === PENDING_COMPANY_VERIFICATION
  • Ouvre le formulaire de validation (§4.4) en modal ou en page dédiée

4.4 Validation d'un Chauffeur

POST /api/company/drivers/{driverId}/validate   Auth : JWT Bearer Token   Content-Type : application/json

Corps de la requête

[
  {
    "name": "Ibrahim Coulibaly",
    "phone": "+22379001234",
    "address": "Rue 45, Bamako",
    "relation": "frère",
    "identityDocumentUrl": "https://cdn.exemple.com/docs/garant1.pdf"
  },
  {
    "name": "Fatoumata Traoré",
    "phone": "+22369005678",
    "address": "Commune V, Bamako",
    "relation": "collègue",
    "identityDocumentUrl": "https://cdn.exemple.com/docs/garant2.pdf"
  }
]
Note : Si les garants existent déjà en base (ajoutés précédemment), envoyer un tableau vide [] est suffisant tant que le total en base est ≥ 2. Les garants soumis remplacent les garants existants (delete + insert).

Réponse succès (200 OK)

{ "message": "Driver validated successfully. Status changed to PENDING_ADMIN_APPROVAL." }

Workflow de validation (étapes UI)

  1. Manager ouvre le dossier du chauffeur
  2. Vérifie que le document d'identité chauffeur est présent (aperçu)
  3. Saisit les informations des 2 garants (formulaire dédié)
  4. Upload les documents d'identité de chaque garant (endpoint upload séparé → URL retournée)
  5. Clique "Valider le dossier"
  6. → Succès : notification de succès + redirection vers la liste
  7. → Erreur : afficher le message d'erreur précis retourné par l'API

Erreurs possibles

HTTPerrorCodeMessageCause
400 MISSING_GUARANTORS "At least 2 guarantors required. Currently has N." Moins de 2 garants en base
400 MISSING_DRIVER_ID_DOC "Driver identity document missing" Aucun document d'identité chauffeur
400 — "Driver status must be PENDING_COMPANY_VERIFICATION. Current status: {X}" Statut incorrect (déjà traité)
403 — "Manager can only validate drivers from their own company" Tentative cross-société
401 — "Unauthorized" Token JWT manquant ou expiré

5. Formulaire Garant

Chaque chauffeur doit avoir exactement 2 garants. Le formulaire de validation contient deux blocs identiques (Garant 1 et Garant 2).

ChampTypeRequisValidation frontend
nameStringOuiNon vide, min 3 caractères
phoneStringOuiNon vide, format téléphone
addressStringOuiNon vide
relationStringOuiNon vide (ex: père, mère, frère, ami, collègue)
identityDocumentUrlString (URL)OuiURL valide après upload
Important — Gestion de l'upload : Le frontend doit d'abord uploader le fichier document via un endpoint d'upload dédié, récupérer l'URL retournée, puis inclure cette URL dans le corps de la requête de validation. Ne pas envoyer le fichier brut dans la requête de validation.

6. Génération du Matricule

Le matricule est généré automatiquement par le backend lors de la validation du chauffeur. Aucune action UI n'est requise.

Format

{PREFIX}-{ANNÉE}-{NNNN}

PREFIX : 3 premiers caractères alphanumériques du nom de la société (MAJUSCULES)
         - Non alphanumériques supprimés
         - Complété par 'X' si nom trop court
ANNÉE  : Année courante (4 chiffres)
NNNN   : ID du chauffeur sur 4 chiffres (zéro-paddé)

Exemples

SociétéID ChauffeurAnnéeMatricule généré
Bamako Livraison Rapide122026BAM-2026-0012
Trans-Mali Express1452026TRA-2026-0145
Sahel Transport & Logistique32026SAH-2026-0003
Gao Transport12002026GAO-2026-1200

8. Récapitulatif des Endpoints

#MéthodeURLAuth requiseDescription
1 GET /api/company/drivers JWT — CM ou SUPPLIER Liste paginée avec filtres (page, limit, status, q)
2 GET /api/company/drivers/pending JWT — CM ou SUPPLIER Chauffeurs en attente de validation (sans pagination)
3 GET /api/company/drivers/{id}/dossier JWT — CM ou SUPPLIER Dossier complet du chauffeur avec garants

10. Gestion des Erreurs

Format de réponse erreur (global)

{
  "status": 400,
  "error": "Bad Request",
  "message": "At least 2 guarantors required. Currently has 1.",
  "path": "/api/company/drivers/12/validate",
  "timestamp": "2026-06-13T10:30:00"
}

Format erreur de validation métier

{
  "errorCode": "MISSING_GUARANTORS",
  "message": "At least 2 guarantors required. Currently has 1."
}

Codes d'erreur métier

errorCodeHTTPMessageAction UI recommandée
MISSING_GUARANTORS 400 "At least 2 guarantors required." Mettre en surbrillance les blocs garant
MISSING_DRIVER_ID_DOC 400 "Driver identity document missing" Indiquer que le doc identité est manquant
— 400 "Driver status must be PENDING_COMPANY_VERIFICATION" Rafraîchir la page (statut déjà changé)
— 403 "Manager can only validate drivers from their own company" Afficher erreur critique, bloquer l'action
— 401 "Unauthorized" Rediriger vers la page de connexion
— 500 "Internal Server Error" Afficher message générique "Erreur serveur"
Bonne pratique : Toujours afficher le champ message de la réponse d'erreur dans l'interface. Utiliser errorCode pour adapter l'affichage contextuel (ex: surligner le bon champ de formulaire).
